### Day 3 — JCS + digest utilities

**Vector architecture (updated 2026-04-20):** spec repo restructured `test-vectors/jcs-parity/` into `inputs/ + generated/ + generate-vectors.mjs`. The generator runs both `canonicalize` and `canonicaljson-go` (via `jcs-parity/go-cli/`) and emits `generated/*.json` with verified `expected_canonical` bytes only when both libraries agree. Our side consumes only `generated/`.

This replaces the earlier Day 3 halt condition — hand-authored expected values were the root cause, not library divergence. The new architecture makes that class of bug impossible.

- [ ] `internal/jcs/` — wrapper around `github.com/gibson042/canonicaljson-go`
- [ ] `internal/digest/` — SHA-256 helpers matching spec's `build-manifest.mjs` behavior
- [ ] **Verify spec-repo generator has been run** — `<spec>/test-vectors/jcs-parity/generated/` should contain all four files (`floats.json`, `integers.json`, `strings.json`, `nested.json`) with `libraries_agree: true` on every case. If `generated/` is empty: run the generator first (`cd <spec>/test-vectors/jcs-parity/go-cli && go build -o jcs-cli && cd .. && node generate-vectors.mjs`). If any case reports `libraries_agree: false`: that's a real upstream divergence — escalate to spec-repo, do not proceed.
- [ ] Consume `<spec>/test-vectors/jcs-parity/generated/*.json` from pinned commit
- [ ] For each case with `libraries_agree: true`: run `canonicaljson.Marshal(case.Input)`, assert output bytes == `case.ExpectedCanonical`
- [ ] For any case with `libraries_agree: false` or missing: fail the test pointing at the spec-repo divergence
- [ ] Green means parity is solved; signing-adjacent conformance work (SV-SIGN-01, SV-CARD-01 signature verification) unblocks
